# Gail's Limited Dataroom Pipeline

Pipeline-first technical case study for Golborne Capital. The project treats the chatbot as a consumer of a normalized dataroom rather than the core product.

## What This Demonstrates

- A unified manual-dropzone ingestion pipeline for UK company datarooms.
- A canonical schema for documents, financial facts, charges, officers, ownership, and events.
- Validation checks for coverage, provenance, and exact factual answers.
- A Streamlit workspace showing the pipeline, document inventory, structured facts, and assistant UI.

## Quick Start

```bash
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
python -m dataroom_pipeline.cli run
streamlit run app/streamlit_app.py
```

The app reads from `data/normalized/dataroom.sqlite`, generated by the pipeline.

## Data Flow

1. Add documents to `data/inbox/`.
2. Register each document in `data/source_manifest.json`.
3. Run `python -m dataroom_pipeline.cli run`.
4. The pipeline writes:
   - normalized text and metadata to `data/processed/`
   - canonical structured facts to `data/normalized/dataroom.sqlite`
   - validation and inventory reports to `reports/`
5. The Streamlit UI reads those outputs.

## Source Strategy

For the Gail's demo, the manifest is pre-populated with the target dataroom document set:

- Companies House overview and filing history for company `06055393`.
- Three years of accounts.
- Charges register and outstanding charge documents.
- Officers and PSC pages.
- Curated news/event sources for expansion, ownership, and lender-risk context.

The first implementation includes a seeded canonical dataset so the UI and assistant run immediately. As source PDFs/pages are added to `data/inbox/`, extracted facts can replace seeded facts without changing the UI.

## Why Manual Dropzone First

The brief rewards the architecture and decision quality more than a brittle crawler. Manual intake keeps the legal/copyright and paywall boundary clear, while still proving the important platform mechanics: provenance, validation, normalization, and repeatability.


## Assistant Runtime And Model Options

The assistant defaults to deterministic structured answers, so no LLM token is used unless explicitly enabled. This protects exact financial and charges answers from hallucination and avoids accidental API spend.

To inspect runtime config:

```bash
python -m dataroom_pipeline.cli llm-status
```

Optional providers are configured through `.env` or deployment secrets:

```bash
ASSISTANT_USE_LLM=false
LLM_PROVIDER=deterministic
```

Free local option with no API key or card:

```bash
ollama pull llama3.2:3b
ASSISTANT_USE_LLM=true
LLM_PROVIDER=ollama
OLLAMA_MODEL=llama3.2:3b
```

OpenAI and Gemini are also supported through `OPENAI_API_KEY` and `GEMINI_API_KEY`, but should be enabled only when intended.

## OpenClaw Integration

This repository includes a workspace OpenClaw skill at `skills/gails-dataroom/SKILL.md`. OpenClaw loads workspace skills from `skills/`, so an OpenClaw agent working in this repo can operate the dataroom through the CLI.

Useful commands for OpenClaw or a terminal demo:

```bash
python -m dataroom_pipeline.cli run
python -m dataroom_pipeline.cli validate
python -m dataroom_pipeline.cli ask "What charges are registered against the company?"
python -m dataroom_pipeline.cli llm-status
```

The Streamlit frontend remains the primary browser demo. OpenClaw is positioned as an agentic operating layer over the same normalized pipeline, not as a replacement for the UI.
